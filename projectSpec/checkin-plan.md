# План: контроль билетов на входе (панель кассира)

Статус: **РЕАЛИЗОВАН** (V01.031, 09.07.2026). Проверен E2E (puppeteer, 14/14 по чек-листу ниже).
Осталось руками: создать пользователя-кассира в «Пользователи» и залогинить его телефон.

## Решения, принятые при обсуждении

- На входе — отдельный человек с обычным смартфоном (интернет есть). Никакого
  оборудования: QR сканируется штатной камерой телефона / любым QR-приложением.
- Роль `CASHIER` уже существует в `src/lib/auth.ts` — кассиру даётся доступ
  ТОЛЬКО к разделу контроля (ни CRM, ни финансов). Сессия живёт 7 дней —
  логин с телефона раз в неделю.
- QR на билете НЕ меняется: та же ссылка `/account/ticket/<id>`, что уже
  разослана в PDF. Гость по ней видит свой билет; сотрудник с ролью — карточку
  контроля. Все ранее проданные билеты работают с первого дня.
- Групповой вход одной отметкой: заказ = одна отметка «использован».
  Частичный проход (пришло 3 из 4) не учитываем — просто пропускаем.
- Светофор кассира:
  - 🟢 оплачен, не использован → кнопка «Пропустить» → usedAt + кто
  - 🔴 уже использован (время + кассир) — копия билета не пройдёт
  - 🟡 не оплачен (new/confirmed) → «Принять оплату на кассе» → статус paid,
    payMethod 'cash' + Transaction(method: 'cash') — наличная выручка честно
    попадает в Финансы отдельно от онлайна → затем «Пропустить»
  - 🔴 заказ отменён / сеанс отменён
- Сознательно НЕ делаем: мобильное приложение, камеру-сканер в браузере,
  пер-местные билеты (серии как у Яндекс Афиши), офлайн-режим.

## Работы

1. **Схема** (`prisma/schema.prisma`, additive): `Booking.usedAt DateTime?`,
   `Booking.usedBy String @default("")` (имя сотрудника). `db push`.
2. **Права**: секция `checkin` в `SECTION_ACCESS` (CASHIER + MANAGER + ADMIN);
   у CASHIER — только она. Сайдбар админки: пункт «Контроль» (для кассира —
   единственный).
3. **Карточка контроля на странице билета**: `/account/ticket/[id]` (или её
   route-обёртка) при staff-сессии с доступом к `checkin` рендерит поверх
   клиентского вида блок: программа, сеанс, состав (взр/дет/льгот + категория
   льготы — «проверьте документ»), имя, сумма, светофор, крупные кнопки под
   палец. Действия — server actions с проверкой роли.
4. **Страница `/admin/checkin`** (mobile-first): сегодняшние сеансы с
   прогрессом «пропущено N из M», поле ручного ввода номера заказа (№1001),
   последние 10 отметок с кнопкой «Вернуть» (откат usedAt).
5. **Телеграм-уведомление** (опционально, если просто): «🎟 Вход: заказ №1001,
   2 взр + 1 дет» — Юлия видит поток входов удалённо.
6. **Пользователь-кассир**: создаётся руками в «Пользователи» (UI уже есть).
   Проверить, что форма позволяет назначить роль CASHIER.

## Проверка (когда будет делаться)

- Скан QR чужим телефоном без логина → обычная страница билета, никаких кнопок.
- Скан с ролью кассира → карточка, «Пропустить» → повторный скан = 🔴 использован.
- Неоплаченный заказ → «Принять оплату на кассе» → появляется в Финансах
  строкой «касса», статус «Оплачена».
- «Вернуть» на /admin/checkin откатывает отметку.
- Дашборд/Финансы: касса и онлайн видны раздельно, суммы сходятся.
